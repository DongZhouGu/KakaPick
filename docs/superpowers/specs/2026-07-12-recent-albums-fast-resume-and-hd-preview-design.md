# BurstPick 最近相册、快速恢复与高清预览设计

## 目标

同时解决三个直接影响日常使用的问题：选片主画面因固定 640px 缩略图而模糊；首页无法快速重新打开最近使用的照片目录；重新选择未变化的目录仍会重复读取 EXIF、提取 RAW 预览和计算感知哈希。

成功标准是：Retina 屏上的选片预览由浏览器按显示尺寸选择足够清晰的资源；首页展示最近 8 个本地相册；点击未变化的历史相册只进行必要的文件清点，然后直接恢复评分、分组与历史。

## 已确认的根因

- `PhotoStage` 不论单张或多张布局都调用默认 `thumbnailUrl(photo.id)`，固定请求 640×640。CSS 会把图片扩展至远大于 640 个 CSS 像素，Retina 屏还需要约两倍设备像素。
- 100% 查看固定请求 1600×1600，它不是原图级预览。
- 目录选择器返回的 `selectionId` 只保存在当前服务进程的 Map 中，使用一次后删除，不能作为最近相册记录。
- `scanAlbum` 在加载旧会话前已经完成 inventory、metadata 和 hashing；旧会话只在末尾恢复评分和分组，因此未变化相册仍承担最昂贵的工作。

## 高清预览

- `thumbnailUrl` 继续生成单个尺寸 URL，新增 `thumbnailSrcSet(photoId, sizes)` 生成宽度描述符。
- 选片舞台提供 640、1280、2048、3200 四档 `srcset`，并根据移动端与每屏数量设置 `sizes`。
- 胶片带和所有组总览继续使用低清固定缩略图，避免无意义的解码与缓存成本。
- 100% 查看请求 4096px 上限版本。服务端现有查询验证已经允许最大 4096，不增加新接口。
- 服务端仍使用 `fit: inside` 和 `withoutEnlargement: true`，不会把低分辨率 RAW 内嵌预览虚假放大。
- 图片加载失败继续遵循浏览器回退与现有错误显示；高清资源不影响评分和导航。

## 最近相册注册表

新增应用数据文件 `recent-albums-v1.json`，与会话文件同处 BurstPick 应用数据根目录，目录权限 0700、文件权限 0600。每条内部记录包含：

- `id`: 规范化绝对路径的 SHA-256。
- `name`: 目录 basename，仅用于展示。
- `canonicalPath`: 仅保存在服务端文件中，绝不进入公开 API。
- `lastOpenedAt`: ISO 时间。

注册表最多保留最近 8 条，按最后打开时间倒序。成功验证目录后更新记录；不存在或无法访问的目录在用户点击时返回安全错误，不在响应或错误中暴露绝对路径。

公开 API 新增：

- `GET /api/v1/albums/recent`：返回 `{ id, name, lastOpenedAt }[]`。
- `OpenAlbumRequest` 新增 `{ recentId }` 分支；服务端从注册表解析路径，再执行与手动路径相同的 `validateDirectory`。

首页在加载时获取最近相册。卡片显示文件夹名和最近打开时间；点击后调用现有 `onOpen({ recentId }, name)`。历史加载失败不阻塞“选择照片文件夹”和手动路径，保持首页可用。

## 快速恢复

扫描仍必须规范化根路径、校验目录、检查缓存位置并完成 inventory，以发现新增、删除、大小或 mtime 变化。inventory 配对完成后立即加载旧会话，并计算当前 fingerprint。

若旧会话满足以下全部条件，则进入快速路径：

- `sourcePathHash` 与当前目录一致。
- `inventoryFingerprint` 与当前清单一致。
- 旧照片 ID 集合与本次配对结果一致。
- 未显式传入不同的分组灵敏度。

快速路径直接返回旧照片、分组、边界覆盖、评分和撤销历史；跳过 metadata、RAW 预览、hashing 和 grouping，只更新时间并重新保存。进度仍依次发出 metadata、hashing、grouping 的已完成状态，避免前端状态机分叉，但这些阶段不会执行昂贵适配器调用。

若 fingerprint 或照片集合变化，则使用现有完整扫描路径；稳定 photo ID 的评分继续恢复，分组按当前清单重算。快速路径返回可从当前 inventory 得出的配对警告和已有照片状态中的拍摄时间回退警告；适配器历史失败警告不会被伪造。

## 组件与边界

- `RecentAlbumsStore`：独立负责注册表验证、原子保存、权限和最多 8 条裁剪。
- `createApp`：暴露 recent API，并在打开路径、选择器或历史项时统一验证并更新注册表。
- `Welcome`：只负责最近相册展示与点击，不接触绝对路径。
- `useAlbum`：加载最近相册并复用现有打开状态机。
- `scanner`：将旧会话加载和 fingerprint 比较提前，封装快速返回路径。
- `api` / `PhotoStage` / `CullingWorkspace`：提供响应式图片 URL 和高清查看。

## 错误与安全

- 注册表损坏时隔离为 `.corrupt-*` 并返回空历史，不能阻止应用启动。
- 最近目录失效时显示“无法访问该最近文件夹，请重新选择”，不泄漏存储路径。
- recent ID 必须是 64 位小写十六进制且只能通过注册表解析。
- 最近 API 不返回绝对路径、源文件名列表或会话内容。
- 注册表采用临时文件、fsync 和无中间半写状态的原子 rename。
- 快速恢复只能在完整 fingerprint 相等时发生，不能用目录 mtime 等弱信号跳过扫描。

## 测试

- `thumbnailSrcSet` 的尺寸排序、编码和宽度描述符；舞台 `srcset`/`sizes` 与 100% 的 4096 URL。
- 注册表保存、排序、去重、裁剪、权限、损坏隔离和不向公开结构暴露路径。
- recent API 的列表、打开、失效目录、非法 ID 和路径不泄漏。
- unchanged fingerprint 第二次扫描不调用 metadata、preview、hash 或 grouping 相关适配器，并完整恢复会话。
- inventory 改变时仍走完整扫描并保留稳定评分。
- Welcome 历史加载失败时常规选择入口仍可用；点击历史调用 recentId。
- 桌面与移动 E2E 检查最近相册入口、快速重新打开和响应式预览属性。
- 全量测试、类型检查、lint、构建、E2E 和元数据安全 smoke 保持通过。

## 非目标

本次不读取 Lightroom 目录缓存、不生成独立代理文件、不绕过目录 inventory、不存储云端历史、不显示完整绝对路径，也不增加最近相册删除或固定功能。
